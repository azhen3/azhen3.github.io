Cisco User Permissions
Administrator Permission
Goal

Implemented a simple user interface for admins to manage permissions of their tools.

Client

Cisco Systems

Achievements

  • Fast: Reduced time from 15 minutes to 5 seconds to give a someone access to a tool.
  • Reduced human error: Instead of memorizing reference ID numbers in the database tagged with the person, the admin can now just allow a user permission from a simple search.
  • Simple: Only the necessary functionalities (add, search, delete) were implemented across two pages.

The Problem

Cisco's collaboration learning platform also handles administration tasks, such as device ordering and content management. Admins needed a quick way to add and remove permissions to these tools.

Previously, the development team would handle permissions to certain users. This process required four different databases to give someone access to an admin tool, taking 15 minutes just to add one person. Instead, the team needed to optimize this process and give admins the ability to manage permissions themselves.

Design Goals

In addition to designing a simple and easy platform for admins, I also turned the requirements into the following goals:

  • The new interface should automate the back-end work required to handle user IDs across multiple databases.
  • This platform should give the service team the ability to grant and remove permission to specific administrator apps.
  • The interface should be visually consistent with the rest of the admin apps, since the service team will be using majority of them.

Early Workflow

One of the biggest priorities was to manage permissions efficiently and quickly. The first iteration consisted of two views: groups and users. Groups included the list of all the admin platforms, and users included the list of people who had permissions. The screen below was the user view, which could expand to show all the platforms a user had access to.

When it was tested with the service team, we realized very quickly that the user view was not as useful as the group view. The admins were more comfortable using the group view and adding permissions from there.

Challenges

As our team collaborated more with the admin users, we discovered the opportunity to incorporate different levels of permissions. For example, a master admin could manage any level of access, while an average admin could only be given access but not be able to grant any.

I iterated several workflows with this in mind, and the designs turned out more complex than originally thought. The team decided that as a first phase release the platform will be simple and only have one level of permissions.

Solution

The final design consisted of only the group view, with an additional page for the specific admin platform. Users of this page can see all the available platforms and the admins who have access to each one. They can also add and remove admins, giving them the ability to manage permissions without the manual help from the engineering team.
